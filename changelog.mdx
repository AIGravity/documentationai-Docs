## Latest verified changes

The current repository snapshot updates implementation details that affect how contributors run, package, and describe Gravity services. These notes summarize verified behavior only; they do not describe planned product UI or endpoint changes.

<Update label="Unreleased" description="Verified implementation notes" tags={["improvement", "documentation"]}>

### Web routes and starter metadata

- The Remix web app currently serves both `/` and `/home`.
- Both routes render the same starter welcome UI and use the same default metadata: title `New Remix App` and description `Welcome to Remix!`.
- The visible UI still includes React Router starter branding, documentation links, and starter copy, so these routes should not be treated as finalized Gravity product branding.

### Versioning and release commands

Each service now has its own version file:

- `apps/web/VERSION`
- `apps/api/VERSION`
- `apps/ai-service/VERSION`

Use the root package commands to bump service versions by release type:

```bash
pnpm version:web:patch
pnpm version:web:minor
pnpm version:web:major
pnpm version:api:patch
pnpm version:api:minor
pnpm version:api:major
pnpm version:ai:patch
pnpm version:ai:minor
pnpm version:ai:major
```

The direct script accepts `web`, `api`, or `ai-service` as the service name:

```bash
./scripts/version-bump.sh web patch
./scripts/version-bump.sh api minor
./scripts/version-bump.sh ai-service major
```

### Docker and port behavior

- The web image exposes port `3000` and runs `npm run start`; local Vite development commonly uses port `5173`.
- The API service listens on port `3001`, but its Dockerfile currently exposes port `3000`. Treat this as a documented mismatch until the Dockerfile or runtime mapping changes.
- The AI service exposes and runs on port `8001` in both development and containerized execution.

### Service naming caveat

The repository still mixes `ai-service` and `py-service` naming in user-visible scripts and metadata. The service directory and version-bump script use `apps/ai-service` and `ai-service`, while the package name remains `@repo/py-service` and the root `dev:py` script still points to `apps/py-service`, which does not exist.

### API and AI endpoint shapes

This update does not change the API or AI service endpoint shapes. The current verified health endpoints remain:

- API: `GET /health` returns `status` and `service` fields.
- AI service: `GET /health` returns a `status` field.

</Update>

## How to read this changelog

Read the newest entry first and treat caveats as current implementation constraints. Entries focus on user-facing behavior, commands, routes, ports, and documented limitations rather than internal refactors.
