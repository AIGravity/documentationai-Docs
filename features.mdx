## Overview

Gravity provides a production-ready monorepo starter with a polyglot stack, shared tooling, and CI/CD so you focus on product code instead of infrastructure scaffolding.

## Polyglot application stack

The core of Gravity is a polyglot stack that models a typical modern product: a web frontend, a backend API, and a dedicated AI service.

### Remix web app (`apps/web`)

The web app uses Remix with Tailwind and Vite for fast, file-based routed experiences.

- Handles user-facing UI and page routing.
- Talks to the backend API for data and actions.
- Shares common tooling with the rest of the monorepo (linting, formatting, TypeScript config).

### NestJS API on Bun (`apps/api`)

The API service runs NestJS on Bun and exposes HTTP endpoints to the web app and external clients.

- Listens on port `3001` in development.
- Enables CORS for the origin `http://localhost:3000`, matching the web app.
- Exposes a health check at `GET /api/health` that returns a JSON status payload.

### FastAPI AI service (`apps/ai-service`)

The AI service uses FastAPI to encapsulate AI-related logic behind a simple HTTP interface.

- Listens on port `8001` in development.
- Exposes a health check at `GET /health`.
- Provides a clear place to integrate models, vector stores, or external AI providers without polluting the core API.

<Callout kind="tip">
  Treat the AI service as an internal dependency of your product. Keep prompts, model config, and experimentation isolated here so the Remix app and NestJS API stay focused on core product behavior.
</Callout>

## Turborepo and pnpm workspaces

Gravity uses pnpm workspaces and Turborepo to coordinate installs, builds, and development across all services.

### Workspace layout

The repository groups applications under `apps/` and shared packages under `packages/` (when present), all managed by pnpm.

- Shared dependencies install once and are reused across services.
- Turborepo understands task relationships and runs tasks in the correct order.
- Caching speeds up repeated builds and tests as the project grows.

### Common commands

Use the workspace scripts to bootstrap dependencies and run all services together.

<CodeGroup tabs="Bootstrap,Dev servers">

```bash
# Install dependencies and bootstrap the workspace
pnpm setup

# Alternatively, run the bootstrap shell script
./scripts/bootstrap.sh
```

```bash
# Start all core services (web, API, AI service) in development
pnpm dev
```

</CodeGroup>

After you run `pnpm dev`, you should see logs for each service starting:

- The web app on `http://localhost:3000`
- The NestJS API on `http://localhost:3001`
- The FastAPI AI service on `http://localhost:8001`

If all services start successfully, you have a full product-like stack running locally.

## Service health endpoints

Each service exposes a simple health endpoint so you verify that it is up and correctly wired.

### API health (`apps/api`)

The NestJS API exposes a health controller that responds on a namespaced path.

```bash
curl http://localhost:3001/api/health
```

Example response:

```json
{
  "status": "ok",
  "service": "api"
}
```

Use this endpoint for local checks, load balancer health probes, or uptime monitors.

### AI service health (`apps/ai-service`)

The FastAPI AI service exposes a plain `/health` endpoint.

```bash
curl http://localhost:8001/health
```

Example response:

```json
{
  "status": "ok"
}
```

<Callout kind="info">
  Standardizing health endpoints across services makes it easier to plug Gravity into your infrastructure tooling, including orchestrators, gateways, and monitoring systems.
</Callout>

## Versioning and CI/CD

Gravity treats each service as a separately versioned unit while still benefiting from the monorepo workflow.

### Per-service semantic versioning

Each service tracks its own semantic version in a `VERSION` file. The repository includes helper scripts that bump versions by service and by level.

Typical commands follow this pattern:

<CodeGroup tabs="Example version bumps">

```bash
# Bump the API service by a minor version (for example, 1.3.0 → 1.4.0)
pnpm version:api:minor
```

```bash
# Bump the API service by a patch version (for example, 1.4.0 → 1.4.1)
pnpm version:api:patch
```

</CodeGroup>

<Callout kind="tip">
  Use per-service versions to communicate changes clearly. For example, you can bump the API minor version when you add a new endpoint without changing the web or AI service versions.
</Callout>

### GitHub Actions and Docker publishing

GitHub Actions workflows automate build, test, and Docker image publishing for each service.

At a high level, the workflows:

- Run on pushes and merges to the main branch.
- Build and test the services using the same pnpm and Turborepo tasks you run locally.
- Build Docker images for each service and push them to GitHub Container Registry (GHCR).

This setup gives you:

- Reproducible builds tied to specific commits and versions.
- Ready-to-run images for staging or production clusters.
- A single place to evolve your pipeline as the monorepo grows.

## How the workflows fit together

The real value of Gravity comes from how the services, tooling, and pipelines interact during everyday development.

<Steps>
  <Step title="Bootstrap the workspace" icon="download">
    Run `pnpm setup` or `./scripts/bootstrap.sh` to install dependencies and configure the Turborepo workspace. This gives you a consistent baseline across all apps and packages.
  </Step>
  <Step title="Develop across services" icon="play-circle">
    Use `pnpm dev` to start the Remix web app, NestJS API, and FastAPI AI service together. As you iterate on one service, the others remain available, mirroring how your product behaves in production.
  </Step>
  <Step title="Validate and ship" icon="check-circle">
    Before merging, rely on local health checks and tests. When you push to main, GitHub Actions runs the same tasks, bumps service versions when you trigger the scripts, and publishes updated Docker images to GHCR.
  </Step>
</Steps>

When everything is wired correctly, you move from local change to deployable images with minimal custom plumbing.
