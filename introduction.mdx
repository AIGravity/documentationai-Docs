Gravity is a polyglot monorepo starter kit, not a hosted SaaS. Clone the repo, run it locally, and customize the services, CI, and infra to match your stack.

</Callout>

## What Gravity is

Gravity is a monorepo that gives you a working full-stack baseline:

- **apps/web**: Remix (React + Tailwind + Vite) on port `3000`
- **apps/api**: NestJS running on Bun with CORS for `http://localhost:3000` on port `3001`
- **apps/ai-service**: FastAPI (Python) on port `8001`

The repo uses **pnpm workspaces** and **Turborepo** to orchestrate installs, builds, and dev workflows across all services.

<Callout kind="tip">

Start with this page to understand the architecture, then move to the [Quickstart](/quickstart) to run Gravity locally and the [Features](/features) page to explore capabilities in depth.

</Callout>

## Architecture at a glance

<Tabs>
  <Tab title="Web app" icon="monitor">

    ### Web app: apps/web {#web-app}

    The web app is a Remix project using React and Tailwind, with local dev served on `http://localhost:3000`.

    - Talks to the API at `http://localhost:3001/api/...`
    - Can call AI routes indirectly via the API or directly to `http://localhost:8001` during development
    - Uses the shared monorepo toolchain for linting, build, and tests

  </Tab>
  <Tab title="API service" icon="terminal">

    ### API service: apps/api {#api-service}

    The API service is a NestJS application running on Bun, exposed on `http://localhost:3001`.

    - CORS is enabled for origin `http://localhost:3000`
    - Health check: `GET /api/health` returns JSON like `{"status":"ok","service":"api"}`
    - Acts as the central backend for web, and as an integration layer to the AI service and external systems

  </Tab>
  <Tab title="AI service" icon="cpu">

    ### AI service: apps/ai-service {#ai-service}

    The AI service is a FastAPI app running on `http://localhost:8001`.

    - Health check: `GET /health` returns JSON like `{"status":"ok"}`
    - Designed for model-backed endpoints, embeddings, or other Python-centric workloads
    - Can be called from the API or directly from local tools during experimentation

  </Tab>
  <Tab title="Tooling & workflows" icon="settings">

    ### Monorepo tooling {#monorepo-tooling}

    Gravity uses pnpm workspaces and Turborepo to standardize workflows:

    - `pnpm setup` bootstraps the workspace
    - `pnpm dev` runs all core services concurrently via Turborepo
    - Shared scripts keep CI and local workflows aligned

  </Tab>
</Tabs>

## Run Gravity locally

Use these steps to clone the repository and bring all services up for local development.

<Steps>
  <Step title="Clone the repo" icon="download">

    Clone `AIGravity/core` to your local machine.

    ```bash
    git clone git@github.com:AIGravity/core.git
    cd core
    ```

  </Step>

  <Step title="Bootstrap dependencies" icon="package">

    Install dependencies and prepare the workspace with pnpm.

    <CodeGroup tabs="pnpm,Shell script">
      ```bash
      pnpm setup
      ```
      ```bash
      ./scripts/bootstrap.sh
      ```
    </CodeGroup>

  </Step>

  <Step title="Start all services" icon="play-circle">

    Run all core apps (web, API, AI service) in dev mode using Turborepo.

    ```bash
    pnpm dev
    ```

    Successful startup shows log output for each service with ports:

    - Web: `http://localhost:3000`
    - API: `http://localhost:3001`
    - AI service: `http://localhost:8001`

  </Step>

  <Step title="Verify health checks" icon="check-circle">

    Confirm each service is responding.

    <CodeGroup tabs="API health,AI service health">
      ```bash
      curl http://localhost:3001/api/health
      # {"status":"ok","service":"api"}
      ```
      ```bash
      curl http://localhost:8001/health
      # {"status":"ok"}
      ```
    </CodeGroup>

  </Step>
</Steps>

<Callout kind="info">

Gravity expects pnpm, Bun, and Python with uv-compatible tooling installed on your machine. See [Quickstart](/quickstart) for exact version recommendations and troubleshooting tips.

</Callout>

## Versioning and CI/CD {#versioning-ci}

Gravity tracks versions per service and ships Docker images via GitHub Actions.

- Each app has a `VERSION` file that defines its semantic version.
- CI pipelines read these versions, build Docker images on merges to `main`, and publish to GitHub Container Registry (GHCR).
- You can extend the workflows to push to your own registries or deployment targets.

This structure keeps service-level changes isolated while still benefiting from a shared monorepo.
