## Prerequisites

<Callout kind="info" title="Prerequisites">

Set up your local environment before you run the Gravity monorepo:

- Node.js 18 or later
- `pnpm` installed globally
- Bun installed and on your `PATH`
- Python 3.11+ and `uv` installed and on your `PATH`
- Git access to the `AIGravity/core` repository

</Callout>

## Start all services

Run the full stack locally: web app, API, and AI service.

<Steps>
  <Step title="Clone the repository" icon="download">

    Use Git to clone the core monorepo and move into it:

    ```bash
    git clone git@github.com:AIGravity/core.git
    cd core
    ```

    You should now see `apps/` and `packages/` directories in the repo root.

  </Step>

  <Step title="Bootstrap dependencies" icon="package">

    Install all workspace dependencies for Node, Bun, and Python in one step.

    <CodeGroup tabs="pnpm setup,Shell script">
    ```bash
    pnpm setup
    ```
    ```bash
    ./scripts/bootstrap.sh
    ```
    </CodeGroup>

    Use `pnpm setup` during day‑to‑day development. The bootstrap script is equivalent and useful for CI or fresh environments.

  </Step>

  <Step title="Run the dev environment" icon="play-circle">

    Start all services via Turborepo from the repo root:

    ```bash
    pnpm dev
    ```

    This command runs the web app, API, and AI service concurrently. Wait until the logs show each service listening on its port with no errors.

  </Step>

  <Step title="Verify each service" icon="check-circle">

    Confirm that every service is up by hitting its health endpoint.

    ```bash
    # Web (browser)
    open http://localhost:3000

    # API health
    curl -i http://localhost:3001/api/health

    # AI service health
    curl -i http://localhost:8001/health
    ```

    Expect `200 OK` responses from `/api/health` and `/health`, and a usable UI at `http://localhost:3000`.

  </Step>
</Steps>

## Run services individually

Run a single service when you do focused development or debugging.

<Tabs>
  <Tab title="Web app only" icon="monitor">

    From the repo root, start the web app:

    ```bash
    pnpm --filter @repo/web dev
    ```

    - URL: `http://localhost:3000`
    - Use this when working on the frontend only, assuming the API and AI service are already running elsewhere.

  </Tab>

  <Tab title="API only" icon="code">

    Start the API service:

    ```bash
    pnpm --filter @repo/api dev
    ```

    - Base URL: `http://localhost:3001`
    - Health endpoint: `GET http://localhost:3001/api/health`

    The API is configured with CORS to allow origin `http://localhost:3000` with credentials enabled, so you can call it directly from the local web app.

  </Tab>

  <Tab title="AI service only" icon="terminal">

    Start the Python AI service:

    ```bash
    pnpm --filter @repo/py-service dev
    ```

    - Base URL: `http://localhost:8001`
    - Health endpoint: `GET http://localhost:8001/health`

    Use this when iterating on AI logic or Python dependencies.

  </Tab>
</Tabs>

## Service map and health checks

Use this as a quick reference for what runs where and how to confirm it works.

| Service    | Purpose      | URL / Port                 | Health endpoint                     |
|-----------:|--------------|----------------------------|-------------------------------------|
| Web        | UI/frontend  | `http://localhost:3000`    | Load root page in a browser         |
| API        | Backend API  | `http://localhost:3001`    | `GET http://localhost:3001/api/health` |
| AI service | AI features  | `http://localhost:8001`    | `GET http://localhost:8001/health`  |

If any health endpoint returns a non‑200 status or times out, check the corresponding dev process logs in your terminal.

## Troubleshooting

<ExpandableGroup>

  <Expandable title="Port already in use (EADDRINUSE)" default-open="false">

  **Symptom**

  - `pnpm dev` or an individual service fails with `EADDRINUSE` or a message like `address already in use` for port `3000`, `3001`, or `8001`.

  **Fix**

  - Stop any existing dev servers using those ports (browser dev tools, other app instances, or orphaned node/python processes).
  - On Unix‑like systems, identify and kill the process:

    ```bash
    lsof -i :3000
    kill <pid>
    ```

  - Restart the relevant command, for example:

    ```bash
    pnpm --filter @repo/web dev
    ```

  </Expandable>

  <Expandable title="Command not found: bun or uv" default-open="false">

  **Symptom**

  - Bootstrap or dev scripts exit with errors such as `bun: command not found` or `uv: command not found`.

  **Fix**

  - Install Bun following the official instructions and ensure `bun` is on your `PATH`.
  - Install `uv` for Python dependency management and confirm `uv --version` works.
  - Re-run:

    ```bash
    pnpm setup
    ```

  </Expandable>

  <Expandable title="API requests blocked by CORS" default-open="false">

  **Symptom**

  - Browser console shows CORS errors when calling the API from the web app.

  **Details**

  - The API enables CORS for origin `http://localhost:3000` with credentials.

  **Fix**

  - Ensure you run the web app from `http://localhost:3000`, not a different port.
  - If you are calling the API from another origin (for example, a separate app), proxy requests through the web app or run your client on `http://localhost:3000`.

  </Expandable>

  <Expandable title="Health endpoints return 500 or timeout" default-open="false">

  **Symptom**

  - `curl http://localhost:3001/api/health` or `curl http://localhost:8001/health` hangs or returns `5xx`.

  **Fix**

  - Confirm the service process is running and not crashing on startup.
  - Stop the dev command with `<kbd>Ctrl</kbd>+<kbd>C</kbd>` and restart:
  
    ```bash
    pnpm dev
    ```

  - Watch the logs for stack traces or missing configuration; address the underlying error, then retry the health check.

  </Expandable>

</ExpandableGroup>
